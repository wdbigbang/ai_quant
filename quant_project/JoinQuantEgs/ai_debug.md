## 一、项目概述

## 二、关键技术要点（调试踩坑记录）

⚠️ **必须遵守**，违反可能导致策略运行失败。

### 2.1 实盘兼容性

#### 1. 避免未来数据（重要踩坑）
盘前数据获取必须使用`context.previous_date`，不能用`context.current_dt`。
**原因**：盘前运行时（如09:25），当天还没有收盘数据，使用`context.current_dt`会导致`get_price()`返回空数据。
**修复**：`get_all_etf_list()`函数中已修复此问题。

#### 2. Panel对象问题
`get_price()`必须加`panel=False`参数。

#### 3. get_current_data()缓存机制
同一定时任务内有缓存，无法覆盖。解决方案：统一获取数据后传递给子函数。

### 2.2 持仓管理

#### 4. 持仓引用问题（重要踩坑）
卖出时保存的持仓信息是引用，下单后会被清空。
**解决方案**：深拷贝关键属性（avg_cost, total_amount, cost_value）。

#### 5. 策略标签系统（重要踩坑）
多策略必须区分持仓归属：
- 买入后设置：`g.strategy_tags[code] = self.strategy_id`
- 卖出后删除：`del g.strategy_tags[code]`
- 卖出前检查：只卖出属于当前策略的持仓

### 2.3 交易日处理

#### 6. 交易日vs日历日
冷静期、资金池调整等时间判断应基于交易日，非日历日。

#### 7. 100股最小单位
买入前检查预期股数，不足100股需调整金额或跳过。

### 2.4 Python语法兼容性

#### 8. 海象操作符问题（重要踩坑）
聚宽平台使用的Python版本不支持海象操作符 `:=`（需要Python 3.8+）。
**问题代码**（第1154行）：
```python
months_since_last_adjustment = (current_year := current_date.year) * 12 + current_month - (...
```
**解决方案**：改为常规赋值语句
```python
current_year = current_date.year
months_since_last_adjustment = current_year * 12 + current_month - (...
```

#### 9. datetime导入问题
文件开头使用`import datetime as dt`，代码中应使用`dt.datetime.strptime()`而非`datetime.strptime()`。

### 2.5 ETF池管理

#### 10. 参考日期选择（重要踩坑）
获取ETF历史数据（如成交量）时，必须使用`context.previous_date`而非`context.current_dt.date()`。
**原因**：盘前获取数据时，当前交易日还没有收盘数据。

#### 11. ETF流动性筛选（重要踩坑）
使用`get_price()`获取成交量数据时，返回的是DataFrame对象，需正确提取volume列。
**验证方法**：
```python
volume_data = get_price(code, end_date=ref_date, count=20, 
                      frequency='daily', fields=['volume'], panel=False)
if volume_data is not None and not volume_data.empty:
    avg_volume = volume_data['volume'].mean()  # 正确提取volume列
```

#### 12. 日志输出精简（重要踩坑）
避免输出过多的DEBUG日志，只保留关键统计信息。
**废弃功能**：ETF池自动获取功能已废弃（v8.5.0），当前使用固定4只ETF池。

### 2.6 全仓模式配置

#### 13. 策略注册逻辑必须统一（重要踩坑）
**问题现象**：当设置 `g.full_ratio_strategy_id = 'test_strategy_1'` 时，系统仍然按照默认资金池分配执行。
**问题原因**：策略注册时没有检查全仓模式配置。
**解决方案**：所有策略在注册时都必须检查 `g.enable_full_ratio_mode` 和 `g.full_ratio_strategy_id`

### 2.7 ETF动量计算优化

#### 14. 线性权重 vs 指数权重（重要踩坑）
**问题现象**：使用指数衰减权重（1-10倍）导致策略收益从198.19%暴跌到-2.58%。
**问题原因**：指数权重衰减幅度过大，导致短期噪音影响被过度放大。
**解决方案**：
- **必须使用线性权重**：`weights = np.linspace(1, 2, n)`（从1到2倍）
- **保持原始参数**：m_days=25, min_score=0, max_score=5, target_num=1
- **使用固定4只ETF池**：518880.XSHG、513100.XSHG、159915.XSHE、510180.XSHG

### 2.8 资金池动态分配

#### 15. 评分系统调用（重要踩坑）
**问题现象**：评分系统代码已实现，但每月日志中看不到评分结果，资金池分配仍使用旧的简单逻辑。
**问题原因**：`allocate_capital_pools`方法中没有调用新的`calculate_strategy_scores`方法。
**解决方案**：在资金分配前添加评分系统调用
```python
# ========== 调用新的评分系统 ==========
log.info("[资金池] 开始计算策略评分（使用新的四维度评分系统）")
strategy_score_results = self.calculate_strategy_scores(context)
```

#### 16. 类型转换错误（重要踩坑）
**问题现象**：运行时出现错误 "unsupported operand type(s) for -: 'dict_values' and 'float'"。
**问题原因**：`sum(final_ratios.values())`返回dict_values对象，不是数值类型。
**解决方案**：添加显式类型转换
```python
total_ratio = sum(float(final_ratios[strategy_id]) for strategy_id in final_ratios.keys())
new_ratio = float(new_ratio)
last_ratio = float(last_ratio)
```

## 三、资金池动态分配系统（v9.0.0）

### 3.1 系统概述

资金池动态分配系统通过四维度评分机制，自动根据策略表现调整资金分配比例，提高整体收益并降低风险。

### 3.2 核心功能

#### 3.2.1 四维度评分系统

| 评分维度 | 权重 | 计算方法 | 评分标准 |
|---------|------|---------|---------|
| 收益率 | 40% | 对数收益率年化处理 | ≥20%:100, ≥10%:80, ≥0%:60, ≥-10%:40, ≥-20%:20, <-20%:0 |
| 胜率 | 25% | 盈利交易占比 | ≥70%:100, ≥60%:80, ≥50%:60, ≥40%:40, ≥30%:20, <30%:0 |
| 风险 | 25% | 最大回撤和波动率 | 综合评分=回撤评分×0.6+波动率评分×0.4 |
| 稳定性 | 10% | 月度收益一致性和连续盈利月数 | 综合评分=标准差评分×0.7+盈利月数评分×0.3 |

#### 3.2.2 资金分配规则

1. **基础分配**：最优策略60-70%，次优策略30-40%
2. **上下限控制**：单个策略资金池不低于20%，不超过80%
3. **平滑过渡**：每次调整最多变化20%，避免剧烈波动
4. **极端情况处理**：连续低分策略暂停分配

#### 3.2.3 调整周期配置

- **调整频率**：每N个月调整一次（默认为1个月，可配置）
- **最低交易日间隔**：20个交易日（可配置）
- **市场停牌检测**：遇到节假日自动顺延调整日期

### 3.3 代码位置

| 功能模块 | 方法位置 | 说明 |

| 评分系统主方法 | ~第2200行 | `calculate_strategy_scores(context)` |

| 胜率评分 | ~第2450行 | `_compute_win_rate_score(performance_data)` |
| 风险评分 | ~第2520行 | `_compute_risk_score(daily_returns)` |
